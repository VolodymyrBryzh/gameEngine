# Roadmap: Процедурна генерація (Chunk System)

## Поточний стан
- [x] Domain Warping (SampleHeight з двома warp-шумами)
- [x] Один меш 512×512 із деревами та камінням
- [x] FPS-контролер із фізикою та бобінгом

## Проблеми поточної архітектури
| # | Проблема | Наслідок |
|---|---|---|
| 1 | Один меш 512×512 = ~3.1M вершин | Неможливо нескінченно масштабувати |
| 2 | `hmap[]` — глобальний масив у RAM | ~1 МБ, не масштабується |
| 3 | Всі об'єкти у плоских векторах | Лінійний пошук без просторового поділу |
| 4 | Фіксований seed `12345` | Один і той самий світ завжди |

## Фаза 1 — Chunk + ChunkManager [РЕАЛІЗОВАНО]
**Файли:** `chunk.h`, `chunk_manager.h`, `main.cpp`

- `Chunk` — 32×32 одиниці, власний меш + об'єкти
- `BuildChunk(cx, cz)` — детермінована генерація по координатах
- `ChunkManager::LoadImmediate` — синхронне завантаження старту (5×5)
- `ChunkManager::Update` — стримінг: 3 нових чанки/кадр, вивантаження за keepRadius
- `ChunkManager::GetHeight` — пряме `SampleWorldHeight` (без кешу, ~12 noise-викликів/кадр)

```
viewRadius = 4   →  ~9×9 = 81 чанків   = ~165k трикутників
keepRadius = 6   →  ~13×13 = 169 чанків максимум в пам'яті
```

## Фаза 2 — Seed + Рандомний старт [РЕАЛІЗОВАНО]
- `PerlinNoise(uint32_t seed)` — параметризований конструктор
- CLI-аргумент: `game.exe --seed 42`
- Seed зберігається у `save/world.dat`, при повторному запуску — завантажується
- Стартова позиція — спіральний пошук зеленої поверхні (h 12..38) від origin
- HUD відображає поточний seed

## Фаза 3 — Виправлення швів між чанками [РЕАЛІЗОВАНО]
- Extended hmap 35×35 замість 33×33 — семплює [-1, CHUNK_SIZE+1]
- Нормалі через центральні різниці: `n = normalize(H(i-1,j)-H(i+1,j), 2, H(i,j-1)-H(i,j+1))`
- Гранична вершина знає висоту сусіднього чанка → шов зникає
- Бонус: smooth shading замість flat — більш природний вигляд рельєфу

---

## Фаза 4 — Асинхронна генерація [РЕАЛІЗОВАНО]
- `BuildChunkCPU` — тільки RAM, безпечна для будь-якого потоку
- Thread pool з 2 воркерами (`std::thread` + `std::mutex` + `std::condition_variable`)
- `workQueue` → воркер → `uploadQueue` → main thread (`UploadMesh` + `LoadModelFromMesh`)
- `inFlight` set запобігає дублюванню задач
- `maxUpload=2` чанки на кадр → без мікрозависань при перетині меж
- `ChunkManager::Shutdown()` коректно зупиняє потоки до `CloseWindow()`

---

## Фаза 5 — LOD [РЕАЛІЗОВАНО]
- LOD 0 (dist ≤ 2): 33×33 = 2048 трикутників + дерева/каміння
- LOD 1 (dist 3–7): 17×17 = 512 трикутників, без об'єктів (4× менше геометрії)
- `viewRadius=7, lodRadius=2, keepRadius=9` — ~225 чанків у пам'яті
- Автоматичний апгрейд LOD1→LOD0 при наближенні (unload + rebuild)
- Нормалі через центральні різниці з кроком step → шви відсутні на обох LOD
